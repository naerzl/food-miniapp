# 行为准则

减少常见 LLM 编码错误的行为准则。将项目特定说明合并到其中。

**权衡：** 这些准则偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。暴露权衡。**

实现前：
- 明确陈述你的假设。如果不确定，请提问。
- 如果存在多种解释，请提出——不要默默选择。
- 如果存在更简单的方案，请指出。有理由时反驳。
- 如果不清楚，请停止。说出困惑所在。提问。

## 2. 简洁优先

**最小代码量解决问题。不写 speculative 代码。**

- 不添加超出需求的特性。
- 不为单次使用的代码创建抽象。
- 不添加未请求的"灵活性"或"可配置性"。
- 不处理不可能发生的场景的错误。
- 如果你能写 200 行但可以用 50 行，重写。

问自己："高级工程师会说这太复杂了吗？"如果是的，请简化。

## 3. 精准修改

**只修改必须改的。只清理自己的烂摊子。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你做法不同。
- 如果注意到无关的死代码，指出它——不要删除。

当你的修改产生孤立代码时：
- 删除因你的修改而不再使用的 imports/variables/functions。
- 不要删除预先存在的死代码，除非被要求。

检验标准：每行修改都应该直接源于用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到完成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后使其通过"
- "修复 bug" → "编写复现 bug 的测试，然后使其通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，简述计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

明确的成功标准让你能独立循环验证。模糊的标准（"让它工作"）需要持续确认。

---

**这些准则生效的表现：** diff 中不必要的修改减少，因过度复杂导致的返工减少，澄清问题出现在实现前而非出错后。
