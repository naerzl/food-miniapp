# 家庭点菜系统产品逻辑文档

## 1. 引言
### 1.1 项目背景
本项目旨在为家庭或小型朋友圈提供一个便捷的点菜与就餐管理工具。通过微信小程序，客人可以浏览当日菜单、点菜并完成支付；厨师通过Web后台管理菜品、查看订单并跟踪状态。系统以极简方式满足家庭内部点餐需求，降低沟通成本，增加用餐趣味性。

### 1.2 产品目标
- 实现客人自助点菜、支付（模拟或真实）功能
- 实现厨师对菜品、订单的管理
- 提供清晰的状态流转（已下单→制作中→已完成）
- 保证数据安全，支持多用户（家庭内多客人）
- 为后续扩展（如多人厨师、历史统计）留有余地

## 2. 产品概述
本系统由三部分组成：
- **微信小程序**：客人端，提供菜单展示、点菜、下单、支付、订单查询等功能。
- **Web管理后台**：厨师端，提供菜品管理（增删改查）、订单管理、状态更新、基础数据统计。
- **后端服务**：基于NestJS + MySQL，提供RESTful API，处理业务逻辑与数据持久化。

## 3. 用户角色
| 角色   | 描述                                     | 使用端       |
|--------|------------------------------------------|--------------|
| 厨师   | 系统管理员，负责菜品维护、订单处理       | Web后台      |
| 客人   | 家庭成员或朋友，通过小程序点菜、支付     | 微信小程序   |

> 注：初期仅设一名厨师，客人无需注册，通过微信授权（openId）识别身份。

## 4. 功能需求

### 4.1 客人端（微信小程序）
#### 4.1.1 用户授权与登录
- 首次打开小程序时，获取微信用户昵称、头像，并绑定openId作为唯一标识。
- 无需额外注册，自动创建客人账户。

#### 4.1.2 菜单浏览
- 展示当日可点菜品列表，支持按分类（如凉菜、热菜、主食、饮料）筛选。
- 每个菜品显示：图片、名称、价格、简介、是否售罄。
- 厨师可通过后台设置菜品“今日供应”状态，小程序仅显示供应中的菜品。

#### 4.1.3 点菜与购物车
- 点击菜品可查看详情，并选择份数加入购物车。
- 购物车内可修改份数、删除菜品，实时计算总价。
- 购物车数据保存在本地（或服务端暂存），下次打开小程序保留。

#### 4.1.4 下单
- 从购物车进入确认订单页，可填写备注（如“少辣”）。
- 提交订单后，生成待支付订单，订单状态变为“待支付”。
- 订单包含：订单号、下单时间、菜品列表、总金额、备注、状态。

#### 4.1.5 支付
- **支付方式设计**：由于是家庭内部系统，初期采用“虚拟支付”，点击“确认支付”即视为支付成功，状态变更为“已支付/待制作”。后续可接入微信支付（需商户号）或家庭内部记账模式。
- 支付成功后，向厨师后台发送新订单通知（可通过WebSocket或轮询）。

#### 4.1.6 订单管理
- 我的订单列表展示所有历史订单，按时间倒序。
- 订单详情显示当前状态（待支付、已支付/待制作、制作中、已完成、已取消）。
- 支持取消待支付订单。

#### 4.1.7 个人信息
- 展示微信昵称、头像。
- 可查看累计消费金额、订单数量等简单统计。

### 4.2 厨师端（Web后台）
#### 4.2.1 登录
- 厨师使用账号密码登录（初始账号由系统预设）。
- 可选微信扫码登录（但Web后台一般用账号密码）。

#### 4.2.2 菜品管理
- **分类管理**：增删改菜品分类（如凉菜、热菜、汤类等）。
- **菜品管理**：
    - 菜品列表：展示所有菜品，支持按分类筛选、按名称搜索。
    - 添加/编辑菜品：填写菜品名称、分类、价格、图片（上传）、简介、是否售罄。
    - 删除菜品（软删除，保留历史订单数据）。
    - 批量设置“今日供应”状态。

#### 4.2.3 订单管理
- **订单列表**：
    - 默认展示当天所有订单，可筛选日期、状态。
    - 列表显示：订单号、客人昵称、下单时间、总金额、状态、操作。
- **订单详情**：
    - 查看订单完整信息，包括菜品明细、备注。
    - 状态流转操作：
        - 待支付订单：不可操作（客人可取消）。
        - 已支付/待制作：厨师可点击“开始制作”，状态变更为“制作中”。
        - 制作中：厨师可点击“完成”，状态变更为“已完成”。
        - 已完成：不可再变更。
        - 已取消：由客人取消或厨师取消（需理由）。
- **新订单提醒**：页面顶部显示新订单数量，点击可刷新列表。

#### 4.2.4 数据统计
- 简单的看板：今日订单总数、今日营收、热门菜品Top5。
- 历史数据统计：按日/周/月展示订单量、营收曲线（可选）。

#### 4.2.5 客人管理
- 查看所有客人（openId、昵称、头像、注册时间、累计消费金额）。
- 可禁用某客人（使其无法下单，用于家庭内部管理）。

## 5. 非功能需求
### 5.1 性能
- 小程序页面加载速度 < 2秒。
- API响应时间 < 500ms（平均）。
- 支持并发请求：按家庭规模，最多10人同时使用。

### 5.2 安全
- 使用HTTPS传输。
- 小程序端敏感操作需校验身份（openId）。
- Web后台需登录态校验，密码加密存储（bcrypt）。
- 防止SQL注入、XSS攻击等。

### 5.3 可靠性
- 订单数据不能丢失，关键操作（下单、支付、状态变更）需事务保障。
- 数据库定期备份。

### 5.4 易用性
- 小程序界面简洁，操作直观。
- Web后台布局清晰，状态变更一键完成。

## 6. 业务流程
### 6.1 客人点菜流程
1. 客人进入小程序，浏览当日菜单。
2. 选择菜品加入购物车。
3. 进入购物车，确认份数，点击“去结算”。
4. 填写备注（可选），提交订单 → 生成待支付订单。
5. 在订单页点击“支付”（虚拟支付），状态变为已支付。
6. 厨师在后台看到新订单，开始制作 → 状态变为制作中。
7. 制作完成，厨师点击完成 → 状态变为已完成。
8. 客人可在小程序中查看订单状态更新。

### 6.2 厨师管理菜品流程
1. 登录Web后台。
2. 进入菜品管理，可新增菜品、编辑现有菜品、设置售罄。
3. 每日开始前，可批量设置今日供应哪些菜品。
4. 查看订单，处理待制作订单。

## 7. 数据模型初步设计
### 7.1 主要实体关系
- 用户（User）：客人信息，与订单一对多。
- 菜品分类（Category）：与菜品一对多。
- 菜品（Dish）：包含菜品基本信息。
- 订单（Order）：包含订单头信息。
- 订单项（OrderItem）：订单与菜品的多对多关系，记录数量、单价。
- 支付记录（Payment）：记录支付流水（可选，虚拟支付可简化）。

### 7.2 核心表结构
#### users
- id (int, PK)
- openId (string, 唯一)
- nickname (string)
- avatarUrl (string)
- createdAt (datetime)
- disabled (boolean)

#### categories
- id (int, PK)
- name (string)
- sortOrder (int)
- createdAt (datetime)

#### dishes
- id (int, PK)
- categoryId (int, FK)
- name (string)
- price (decimal)
- image (string)
- description (text)
- available (boolean)  // 是否在售（全局）
- todaySupply (boolean) // 今日是否供应
- createdAt (datetime)

#### orders
- id (int, PK)
- orderNo (string, 唯一)
- userId (int, FK)
- totalAmount (decimal)
- status (enum: 'pending_payment', 'paid', 'preparing', 'completed', 'cancelled')
- remark (text)
- createdAt (datetime)
- updatedAt (datetime)

#### order_items
- id (int, PK)
- orderId (int, FK)
- dishId (int, FK)
- quantity (int)
- price (decimal) // 下单时的单价，快照

#### payments（可选，虚拟支付可省略）
- id (int, PK)
- orderId (int, FK)
- amount (decimal)
- paymentMethod (string) // 'virtual', 'wechat'
- status (string) // 'success', 'failed'
- createdAt (datetime)

## 8. 技术栈建议
### 8.1 后端
- **框架**：NestJS (Node.js) – 提供模块化架构，易于维护和扩展。
- **数据库**：MySQL 8.0 – 关系型数据库，适合结构化数据。
- **ORM**：TypeORM 或 Prisma – 与NestJS集成良好。
- **接口文档**：Swagger (通过NestJS装饰器自动生成)。
- **认证**：JWT (Web后台登录)，小程序使用openId作为身份凭证。
- **实时通知**：WebSocket (NestJS Gateway) 或轮询，用于新订单提醒。

### 8.2 前端（小程序）
- **框架**：微信小程序原生开发 或 使用Uniapp（考虑到只有微信，原生更轻量）。
- **状态管理**：小程序自带全局数据或使用Vuex（如果用Uniapp）。

### 8.3 前端（Web后台）
- **框架**：React 18 + Vite – 快速构建，开发体验好。
- **UI组件库**：Ant Design 或 TailwindCSS + 自定义组件（用户要求TailwindCSS）。
- **状态管理**：Zustand 或 Redux Toolkit。
- **路由**：React Router v6。
- **HTTP请求**：axios。

### 8.4 部署
- **后端**：使用Docker容器化，部署在云服务器或家庭NAS。
- **数据库**：MySQL同样容器化或独立安装。
- **前端**：Web后台构建后部署在Nginx静态服务；小程序上传至微信平台审核。

## 9. 迭代规划建议
### 第一阶段（MVP）
- 实现客人端基本点菜、下单、虚拟支付
- 实现厨师端菜品管理、订单状态流转
- 完成基础数据模型与API

### 第二阶段
- 接入真实微信支付（如需要）
- 增加数据统计与看板
- 优化用户体验，加入通知提醒（订阅消息）

### 第三阶段
- 支持多厨师角色与权限细分
- 增加家庭多人共享点菜（如多人同时点菜合并订单）
- 增加菜品评价功能
